## 3. 环境搭建与启动

### 3.1. 安装依赖

进入项目根目录，安装所有依赖：

```bash
pnpm install
```

### 3.2. 数据库配置

**核心要点**: 确保 `.env` 文件中的 `DATABASE_URL` 配置正确，且**指向 `test` 数据库**。这是系统所有数据操作的唯一来源。

- **文件**: 项目根目录下的 `.env`。
- **内容**: 

```env
DATABASE_URL=mysql://paZkiNgy2nHQcsT.root:mB5jFs2uVaZjEegW@gateway01.ap-southeast-1.prod.aws.tidbcloud.com:4000/test?ssl={"rejectUnauthorized":true}
```

> **注意**: `ssl` 参数的值是一个 JSON 字符串，**必须**使用双引号 `"` 包裹，且内部的引号需要转义 `\"`。错误的格式会导致数据库连接失败，数据无法加载。

### 3.3. 启动项目 (开发模式)

项目启动后，前端和后端服务将同时运行，并自动从 `.env` 文件加载 `DATABASE_URL`。

```bash
pnpm dev
```

- **端口**: 默认在 `3000` 端口启动。如果端口被占用，系统会自动尝试其他端口。
- **登录**: 开发模式下，如果未配置 OAuth 环境变量，系统会自动注入一个 **演示用户 (ID=8, `admin@shenyun.com`)**，无需手动登录即可访问所有功能。

### 3.4. 数据库迁移

如果 `drizzle/schema.ts` 文件有变更，需要执行数据库迁移：

```bash
pnpm db:push
```

## 6. 常见问题与解决方案

### 6.1. 数据库连接失败 / 数据不显示

- **问题现象**: 页面显示"暂无数据"或"加载中"，控制台报错数据库连接失败。
- **原因**: `DATABASE_URL` 环境变量未正确设置，或 `ssl` 参数格式错误。
- **解决方案**: 
    1. **检查 `.env` 文件**: 确保项目根目录下存在 `.env` 文件，且 `DATABASE_URL` 完整且正确，特别是 `ssl={\"rejectUnauthorized\":true}` 部分的转义。
    2. **使用 `pnpm dev` 启动**: 确保使用 `pnpm dev` 命令启动项目，该命令会自动加载 `.env` 文件中的环境变量。
    3. **检查服务器日志**: 查看服务器启动日志 (`/tmp/erp.log` 或控制台输出)，确认数据库连接是否成功。

### 6.2. 端口被占用

- **问题现象**: 服务器启动失败，提示端口已被占用。
- **原因**: 之前启动的进程未完全关闭，或有其他应用占用了 3000 端口。
- **解决方案**: 
    1. **查找并杀死进程**: 使用 `fuser -k 3000/tcp` (如果知道端口) 或 `ss -tlnp` 结合 `kill -9 <PID>` 杀死占用端口的进程。
    2. **重新启动**: 确保端口释放后，再次运行 `pnpm dev`。

### 6.3. 前端页面显示"加载中"或数据不刷新

- **问题现象**: 页面数据长时间不显示，或修改数据后页面未更新。
- **原因**: 
    1. **后端 API 错误**: 后端 API 返回数据格式不正确或报错。
    2. **前端缓存**: 浏览器缓存导致旧数据。
    3. **tRPC 参数验证**: 前端传递的参数与后端 tRPC 路由的 `zod` 验证不匹配 (例如 `undefined` 被序列化为 `null` 导致验证失败)。
- **解决方案**: 
    1. **检查后端 API**: 直接通过 `curl` 命令测试后端 API (`http://localhost:3000/api/trpc/<router>.<procedure>`)，确认返回数据是否正确。
    2. **清除浏览器缓存**: 尝试硬刷新页面或清除浏览器缓存。
    3. **检查 tRPC schema**: 对比前端传参和后端 `routers.ts` 中 `zod` schema 的定义，确保类型匹配。必要时使用 `z.string().nullable().optional()` 兼容 `null` 值。
